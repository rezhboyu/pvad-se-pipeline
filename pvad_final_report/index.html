兩者都跨過三項目標。Ensemble 主要收益不在 F1，而在 FAR：從 0.080 → 0.054，誤報少 32%，TAR 也微升 0.006。F1 邊際增益是 +0.015，給的是「最後一哩」。

Recipe — 兩配置的計算公式

# Per 0.5s segment GRU_alone = zscore( mean4ms-frames( p[1] − (p[0] + p[2] + p[3]) ) ) # 連續分數，不是 binary rule p[1] > rest (那只有 F1=0.84) # pred = GRU_alone ≥ -0.05 ENS_4vec = 0.5·z(cos_target) + 0.5·z(cos_target − max(cos_cohort)) + 2.0·z(GRU) # pred = ENS_4vec ≥ 0.05 # 4D weight sweep (5×5×5×5) 後 asnorm 權重收斂到 0，被 GRU 訊息吸收

Deployment — 兩個方案怎麼選

GRU alone

0.74 MB · 189K params
F1 0.9235 · TAR 0.926 · FAR 0.080

選它如果：手機 / 邊緣裝置、模型大小是首要、不想跑 CAM++（28 MB）。已過三項目標。

注意：必須用 p[1]−rest 連續分數 + 學習閾值，不要用論文預設的二元規則（後者只有 F1=0.84）。

GRU × CAM++ Ensemble

29 MB total
F1 0.9386 · TAR 0.932 · FAR 0.054

選它如果：要最低 FAR（誤報少 32%）、有後端推理算力、特別在 fan noise / SNR=1 等惡劣條件需要邊際提升。

多消耗：+28 MB CAM++、+150 KB cohort、+CAM++ 推理 30ms/0.5s。
