所有演算法 (分類)

① Fbank bug 是過去評估失準的根源
HANDOVER 的 F1=0.7255 基線是用錯誤 fbank (32ms FFT 全長，無 CMN) 跑出。換成 Kaldi 標準後，centroid cos(spk4,spk7) 從 0.62 降到 0.095，所有方法才有意義。
② 對齊抽取 (利用 0.5s 邊界) 是最大 wins
sliding F1=0.74 → aligned A0 F1=0.79 (+0.05) → multi-chunk A3 F1=0.82 (+0.03)。0.3s 視窗對 0.5s 段而言，邊界 ±0.1s 是污染區，避開後 F1 跳 +0.08。
③ Ensemble (A8) 比單一方法穩定
A8 ensemble F1=0.85，三種錯誤模式互補：baseline 對 threshold 敏感、AS-norm 對 cohort 偏置敏感、discriminative 對 max cohort 敏感。融合後 robust。
④ P2 Force-Alternation 突破天花板，但脆弱
從 chunk-level 決策 F1=0.85 跳到 file-level alternation F1=0.99。代價：完全依賴「嚴格交錯」結構假設，真實對話會崩。
⑤ 提高決策率到 0.1s 不一定提高精度
fixed_0.5s 上：2 dec/sec aligned F1=0.85；10 dec/sec sliding F1=0.64~0.72。
原因：每秒 10 個決策意味每個決策只能根據 1 個 chunk；2 dec/sec 用 3 chunks 平均，精度更高。
⑥ Hysteresis 確實有效 (對 sliding 配置)
在 0.3s sliding × 0.1s hop 上，Hysteresis (M5) F1=0.72 是 10/sec 系列最佳。雙閾值避免邊界 score 震盪造成的 0/1 閃爍。
⑦ 0.5s sliding × 0.1s hop 是錯誤組合
10 個視窗裡 8 個跨 GT 邊界 → cosine 被汙染 → FAR 必爆 (≥0.67)。所有後處理 (hangover, hysteresis, energy filter) 都救不了。

部署建議 (按場景)

★ 推薦：Hybrid Deployment

F1=0.85, 10 outputs/sec
• 模型：A8 ensemble，2 推理/秒（含 3 chunks 同時批次推理）
• 輸出：每個 A8 決策複製到 5 個 0.1s 子段
• 計算量：1 batch / 0.5s = ~30ms 推理
• 時間解析度：100ms (足夠應用)

純通用 PVAD

F1=0.77 (Sliding C)
完全不假設邊界，frame-level 評估。任何結構都能用，但精度較低。

嚴格交錯場景

F1=0.99 (P2)
如語音遊戲、輪流問答系統。需保證真實資料嚴格交錯，否則崩。

Hybrid 部署 spec

[Input]   16kHz mono audio stream

[Step 1]  Buffer 0.5s window
[Step 2]  Extract 3 chunks @ offsets (0.05, 0.1, 0.15)s, each 0.3s
[Step 3]  Fbank (Kaldi Povey + preemph + CMN) → CAM++ batch inference (1×3×24×80)
[Step 4]  Per-chunk score: cos vs target_centroid, vs cohort (200 spk)
[Step 5]  Per-chunk: AS-norm + Discriminative
[Step 6]  Multi-chunk mean for each score type → standardize → ensemble
[Step 7]  Threshold (calibrated per cohort = ~-0.7) → binary decision
[Step 8]  Output: 5× duplicate to 0.1s rate

[Output]  10 binary labels/sec, latency ≈ 500ms (one window)
[Compute] ~30ms CPU per 500ms audio (real-time × 17)
